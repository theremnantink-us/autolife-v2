# Инструкция — проект D (новые админ-фичи на localStorage)

**Дата:** 2026-04-27
**Решение:** Вариант (2) из обсуждения — D строится поверх localStorage с сигнатурами, готовыми под Supabase. Когда дойдём до проекта C, тела функций переключатся на supabase-js без изменения UI.

---

## Что будет сделано в D

Пять под-фич, каждая своим спеком/планом/итерацией:

1. **Дневной журнал начислений** (`salary_accruals`)
   Таблица «сотрудник × день». Начальник вбивает руками сумму, у Ивана/Сергея есть переключатель «админ/мойщик» (даёт `role_kind`). Можно редактировать прошлые дни, добавлять бонус-строки.

2. **Поток выплат** (`salary_payouts` + RPC `close_payout`)
   Кнопка «Выплатить → выбрать сотрудника → выбрать диапазон дат». Программа показывает: сумма начислений, минус авансы и штрафы в окне, итого к выдаче. С разбивкой «за админ-смену / за мойку» для Ивана/Сергея. Подтверждение → запись в `salary_payouts`, начисления и удержания помечаются `payout_id`.

3. **Штрафы** (`deductions` с `type='fine'`)
   Зеркало вкладки «Авансы» — та же форма, переключатель типа. Вычитается из выплаты вместе с авансами.

4. **Редактор `scheduled_payouts`**
   Раздел «Настройки → Даты выплат». Календарик на 12 месяцев, по умолчанию 1/16, можно сдвинуть на конкретный месяц (форс-мажор). На дашборде — «через N дней плановая выплата».

5. **График работы** (`work_schedule`)
   Таблица «сотрудник × день месяца» с дропдауном «админ/мастер/выходной» в каждой ячейке. Чисто визуализация, на ЗП не влияет (как договаривались в Q4-A).

## Что от тебя нужно прямо сейчас

**Ничего.** Можно начинать D без env-ключей и auth — всё работает в браузере на localStorage.

Но если хочешь сразу проверить, что Supabase-клиент собирается (на будущее) — создай `site/.env.local` со строкой:
```
PUBLIC_SUPABASE_URL=https://bqvmhbxwptwuazymqwqk.supabase.co
PUBLIC_SUPABASE_ANON_KEY=<вставь ключ из Supabase Dashboard → Settings → API → "anon public">
```
Это **необязательно** для D, но без этого `astro check` будет ругаться на `src/lib/supabase.ts` (потому что throw при пустых env-ах).

## Что от тебя нужно когда дойдём до C (auth + Supabase swap)

В порядке:
1. `.env.local` с реальным `PUBLIC_SUPABASE_ANON_KEY` (если ещё не).
2. Email-адреса трёх аккаунтов админки:
   - Артур → `role=owner`
   - Ирина → `role=owner`
   - Билл → `role=limited`, привязка к `employee_id='bill'`
3. Готовность пройти magic-link auth по очереди — Supabase отправит письмо, ты кликнешь по ссылке, я зафиксирую `dashboard_users` рядом со свежесозданным `auth.users.id`.
4. Решение по `service_role_key` — он нужен только для серверных endpoints (например, `/api/admin/*` с проверкой роли). Хранится **строго** в `site/.env.local` под именем `SUPABASE_SERVICE_ROLE_KEY` (без `PUBLIC_` — Astro не выкатит его в браузер). Никогда не коммитим.

## Как будет работать миграция данных при свапе на Supabase (проект C)

В `store.ts` на этапе D добавится поле `_storage` каждого entity, но публичный API `appointmentsStore.list/upsert/...` остаётся. В C тело функций переключается на `supabase.from(...)`.

Что произойдёт с тестовыми данными, которые ты введёшь в D на localStorage:
- **По умолчанию**: остаются в localStorage браузера, в Supabase не уезжают.
- **Если хочешь сохранить**: я добавлю в `/admin/settings` кнопку «Импорт из локального хранилища» — одноразовая выгрузка JSON и заливка через `supabase.from(...).insert()` с галочкой «удалить из localStorage после успеха». Скажи, если нужно — учту в плане D.

## Безопасность пока D в работе

- D пишет только в localStorage браузера. Никакие данные не уходят на сервер.
- `/admin` пока **открыт без пароля** (как сейчас) — auth уезжает в C. Не публикуй ссылку и не давай URL посторонним до C.
- Старый PHP-бэк (`api.php`, `busy_dates.php`) продолжает работать на 4321 — мы его не трогаем до проекта B.
